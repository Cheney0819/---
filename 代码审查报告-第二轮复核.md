## 一、🔴 Critical 问题（两轮一致确认）

### 1.1 私钥以 JWK 格式明文存储在 localStorage

**确认方**：Dalton · Anscombe · Franklin · Ampere · Poincare（5/6 全部确认）

**文件**：`packages/web/src/lib/crypto.ts:125-127`

```typescript
const privateKeyJwk = await crypto.subtle.exportKey('jwk', keyPair.privateKey);
localStorage.setItem(`private_key_${userId}`, JSON.stringify(privateKeyJwk));
```

**攻击场景（Ampere 提供）**：
```javascript
// 假设存在一个 XSS 漏洞（如通过 displayName 注入）：
const storedKey = localStorage.getItem('private_key_user123');
const jwk = JSON.parse(storedKey);
// jwk.d 就是完整的 EC 私钥（32 字节秘密标量）
// 攻击者可以用它 ECDH 计算共享密钥，解密所有历史消息
```

**修复方案**：
1. **最优**：私钥仅保存在内存中（CryptoKey 对象），页面刷新后重新从服务端获取
2. **折中**：使用 PBKDF2 从用户密码派生密钥，AES-GCM 加密私钥后存入 localStorage
3. **最低**：迁移到 IndexedDB + 加密层

### 1.2 JWT 密钥强度严重不足

**确认方**：Dalton · Ampere · Poincare（3/6 确认）

**文件**：`packages/api/.env:8`

当前密钥 `shiguangjian-secret-key-2024` 可预测，含年份。

**Ampere 补充**：`getJwtSecret()` 直接用 TextEncoder 编码字符串作为 HMAC 密钥，没有通过 KDF 派生。如果 JWT_SECRET 是弱密码，攻击者可离线暴力破解。

**修复方案**：
```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

### 1.4 端到端加密链路断裂

**确认方**：Poincare（新发现，无争议）

**问题本质**：项目宣称"端到端加密"，但：
- crypto.ts 定义了 ECDH + AES-GCM 加密体系
- 所有 API Route Handler 接收的 content 字段直接写入数据库，未经加密验证
- FastAPI 注释说"客户端发送的内容已经是加密后的密文"，但 Next.js API Routes 和 FastAPI 是**两套独立服务**——Next.js 本身就是 API 入口
- **实际上明文内容直接存储在数据库中**，E2EE 只是客户端侧的 UI 展示

**Ampere 补充**：即使加密链路完整，当前实现也存在：
- 无 HKDF/PBKDF2 密钥派生（裸 ECDH → AES key，存在小子群攻击风险）
- 无前向保密（单 ECDH，私钥泄露则所有历史消息可解密）
- 无密钥认证/身份绑定（中间人可替换公钥）

**修复方案**：
1. 要么真正实现 E2EE（服务端永远不接触明文）
2. 要么明确放弃 E2EE，改用 TLS + 服务端 AES-256-GCM 加密存储
3. 无论哪种，都要引入 HKDF 和 Double Ratchet 算法

## 二、🟠 High 问题（多轮交叉确认）

### 2.1 CORS 配置过于宽松

**确认方**：Dalton · Anscombe · Lovelace · Poincare（4/6 确认）

**文件**：`packages/api/src/index.ts:40-43`

`origin: true` + `credentials: true` = 任何网站都可携带认证信息发起跨域请求。

---

### 2.2 无服务端路由保护（Middleware）

**确认方**：Anscombe（唯一确认，但无争议）

所有认证检查在客户端完成，用户可直接通过 URL 访问受保护页面。

---

### 2.3 权限校验逻辑重复 + 消息删除缺 pair 校验

**确认方**：Franklin · Ampere · Poincare（3/6 确认）

**Franklin**：`OR: [{ userAId: userId }, { userBId: userId }]` 在 10+ 文件中重复。

**Ampere 补充**：`DELETE /api/messages/[pairId]/[messageId]` 只校验了 `message.senderId !== auth.id`，没有校验 pair 成员关系——任何人只要知道 messageId 就能删除消息。

---

### 2.4 大量使用 any 和类型断言

**确认方**：Franklin · Lovelace（2/6 确认）

`websocket.ts` 中 `message: any`，各路由中 `request.user as { id: string }` 出现 10+ 次。

**Lovelace 补充**：以下路由完全不校验输入结构，使用 `as` 类型断言绕过：
- `users.ts:9` — `deviceToken`
- `users.ts:22` — `displayName/avatarUrl`
- `users.ts:65` — `publicKey`
- `pairs.ts:8` — `partnerId`

---

### 2.5 数据库缺少关键索引

**确认方**：Franklin · Poincare（2/6 确认）

`Message.deletedAt`、`TimeCapsule.status`、`SharedDiary.pairId`、`SharedAlbum.pairId` 均无索引。

---

### 2.6 认证逻辑分散且不统一

**确认方**：Franklin · Poincare（2/6 确认）

Fastify 用 `@fastify/jwt`，Next.js 用 `jose`；两套认证机制各自实现。

---

### 2.7 WebSocket 连接泄漏

**确认方**：Franklin · Lovelace（2/6 确认）

`onlineUsers` Map 只清理事件驱动的 close 事件。心跳机制已定义但未实现。

---

### 2.8 配对创建竞态条件（TOCTOU）

**确认方**：Lovelace（新发现，无争议）

**文件**：`packages/api/src/routes/pairs.ts:8-46`

两个并发请求可同时通过"检查是否已配对"步骤，然后都成功创建。虽然有 `@@unique([userAId, userBId])` 约束，但异常未被优雅处理。

---

### 2.9 邀请码 Math.random() — 重要性升级

**确认方**：Dalton · Ampere · Lovelace（3/6 确认）

**第一轮**标记为 Low，**第二轮 Ampere 升级为 Critical**（类比 CVE-2013-4392），**Lovelace 升级为 High**。

4 位邀请码 + Math.random() = 167 万种可能 + 可预测随机数 = 秒级暴力破解。

---

### 2.10 端到端加密架构问题

**确认方**：Poincare · Ampere（2/6 确认）

除了私钥存储，Ampere 还发现：
- 无 HKDF 密钥派生（裸 ECDH → AES key，存在小子群攻击风险）
- 无前向保密（单 ECDH，私钥泄露则所有历史消息可解密）
- 无密钥认证/身份绑定

## 三、🟡 Medium 问题

### 3.1 JWT 30天有效期 — 重要性升级

**确认方**：Dalton · Anscombe · Ampere · Poincare（4/6 确认）

**第二轮 Ampere 补充**：JWT Claims 严重缺失——无 `jti`（无法撤销）、无 `sub`（违反 RFC 7519）、无 `iss`/`aud`（无法锚定）。类比 CVE-2015-9235。

### 3.2 无 CSP 头

**确认方**：Anscombe · Ampere（2/6 确认）

Ampere 提供了 PoC：通过 displayName 注入 `<script>` 可在无 CSP 情况下直接执行。

### 3.3 原型链污染风险

**确认方**：Lovelace · Poincare（2/6 确认）

`z.any()` 跳过校验 + `request.body as Type` 绕过校验 = 攻击者可传入含 `__proto__` 键的对象。

### 3.4 搜索接口无速率限制

**确认方**：Lovelace（新发现，无争议）

`/search` 使用 `LIKE %q%` 无法走索引，高频调用可导致数据库 CPU 飙升。

### 3.5 消息删除缺 pair 校验

**确认方**：Ampere（新发现，无争议）

`DELETE /messages/[pairId]/[messageId]` 只校验 senderId，不校验 pair 成员关系。

### 3.6 双 Schema 冗余

**确认方**：Poincare（新发现，无争议）

`packages/api/prisma/schema.prisma` 和 `packages/web/prisma/schema.prisma` 内容完全相同。

### 3.7 类型定义不一致

**确认方**：Poincare（新发现，无争议）

`packages/shared/types.ts` 与 `packages/web/src/types/index.ts` 是两个不同的类型定义，同一数据在不同地方有不同契约。

### 3.8 Prisma Schema 问题

**确认方**：Poincare（新发现，无争议）

- 无软删除标记
- 无审核字段
- `mediaUrls`/`tags` 类型为 Json 而非数组
- 无 updatedAt 时间戳
- 外键全部 ON DELETE RESTRICT
- String 字段未用 Enum 约束

3. **认证不安全**——弱 JWT 密钥 + 30 天不过期 + 无 claims + 无 refresh token
