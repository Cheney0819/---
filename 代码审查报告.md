## 一、Critical 问题（必须立即修复）

### 1.1 私钥以 JWK 格式明文存储在 localStorage

**涉及文件**：`packages/web/src/lib/crypto.ts:125-127`、`packages/web/src/contexts/EncryptionContext.tsx:44`

```typescript
const privateKeyJwk = await crypto.subtle.exportKey('jwk', keyPair.privateKey);
localStorage.setItem(`private_key_${userId}`, JSON.stringify(privateKeyJwk));
```

**问题本质**：项目宣称"端到端加密"，但核心加密密钥却明文存放在 localStorage。任何 XSS 攻击（包括第三方库注入）均可直接读取所有用户的 ECDH 私钥，解密全部历史消息。所谓 E2E 加密在此场景下形同虚设。

**三方一致判定**：三位审查者均将此列为最高优先级。

**修复方案**：
1. **最优方案**：私钥仅保存在内存中（CryptoKey 对象），页面刷新后重新从服务端获取密钥对
2. **折中方案**：使用用户密码通过 PBKDF2 派生密钥，再用该密钥 AES-GCM 加密私钥后存入 localStorage
3. **最低要求**：迁移到 IndexedDB + 额外加密层，避免 localStorage 被常见 XSS 工具直接读取

---

### 1.2 JWT 密钥强度严重不足

**涉及文件**：`packages/api/.env:8`、`packages/web/src/lib/auth.ts`

当前密钥为 `shiguangjian-secret-key-2024`——可预测、含年份、仅 31 字符。

**问题本质**：攻击者可通过暴力破解或字典攻击伪造任意用户的 JWT Token，完全接管任何账户。Web 端使用 jose 库的 HS256 算法，同样依赖此弱密钥。

**修复方案**：
```bash
# 生成至少 256 位随机密钥
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```
- 迁移到 RS256/ES256 非对称签名算法（公钥可公开验证，私钥仅服务端持有）
- 密钥应通过环境变量注入，绝不硬编码

---

### 1.3 数据库凭据存在于仓库中

**涉及文件**：`packages/api/.env`

`.env` 文件包含明文数据库密码和生产环境完整连接字符串（含密码）。虽然 `.gitignore` 已配置忽略，但文件已在仓库中存在。

**修复方案**：
1. 立即轮换所有数据库密码（开发 + 生产）
2. 确认 `.env` 从未被提交到 Git 历史；如已提交，用 git filter-branch 或 BFG Repo-Cleaner 彻底清除
3. 生产环境凭据通过 CI/CD Secret 管理（Vercel Environment Variables / AWS Secrets Manager）注入

---

### 1.4 两套 API 共存导致架构分裂

**涉及文件**：`packages/api/src/routes/`（Fastify）vs `packages/web/src/app/api/`（Next.js Route Handlers）

**问题本质**：项目同时存在 Fastify 独立 API 服务和 Next.js Route Handlers，两套 API 实现几乎完全相同的 CRUD 逻辑，但路由命名和响应格式有细微差异。这导致：
- 维护成本翻倍
- 认证逻辑分散（@fastify/jwt vs jose）
- 数据库连接池无法共享
- 难以确定"真相来源"

**修复方案**：选择其一作为唯一数据源
- **方案 A（推荐）**：Fastify 作为唯一 Backend API，Next.js 只做 SSR/SSG 和静态页面
- **方案 B**：移除 Fastify，Next.js Route Handlers 作为唯一 API，部署到 Vercel/Cloudflare

## 二、High 问题（应在本迭代内修复）

### 2.1 CORS 配置过于宽松

**文件**：`packages/api/src/index.ts:40-43`

```typescript
await app.register(cors, { origin: true, credentials: true });
```

`origin: true` 配合 `credentials: true` 意味着任何网站都可以携带认证信息发起跨域请求。

**修复**：将 origin 限制为具体可信域名白名单。

---

### 2.2 无服务端路由保护（Middleware）

**现状**：所有认证检查在客户端组件内完成，无 Next.js Middleware。用户可直接通过 URL 访问 `/dashboard`、`/chat/*` 等受保护页面。

**修复**：在 `packages/web/src/middleware.ts` 中添加 Next.js Middleware，在请求级别拦截未认证路由。

---

### 2.3 权限校验逻辑在每个路由中重复

**现状**：`OR: [{ userAId: userId }, { userBId: userId }]` 这种"验证用户拥有某资源的所有权"的逻辑在 10+ 个文件中重复出现。

**修复**：创建 Fastify 插件或 Next.js Middleware 将所有权验证抽象为装饰器/守卫函数。

---

### 2.4 大量使用 any 和类型断言

**现状**：`websocket.ts` 中 `socket.on('message', async (message: any))`，各路由中 `request.user as { id: string }` 出现 10+ 次，`z.any()` 绕过类型安全。

**修复**：定义完整的 Fastify TypeScript 类型声明，使用 FastifyRequest 泛型。

---

### 2.5 数据库缺少关键索引

**文件**：`packages/api/prisma/schema.prisma`

| 字段 | 影响 |
|------|------|
| Message.deletedAt | 软删除消息过滤全表扫描 |
| TimeCapsule.status | 定时任务全表扫描 pending 胶囊 |
| SharedDiary.pairId | 按配对查日记全表扫描 |
| SharedAlbum.pairId | 同上 |

**修复**：在 Prisma schema 中添加 `@@index([...])`。

---

### 2.6 认证逻辑分散且不统一

**现状**：Fastify 用 @fastify/jwt，Next.js 用 jose；两套认证机制各自实现。

**修复**：统一 JWT 验证逻辑，提取为共享库。

---

### 2.7 WebSocket 连接泄漏

**文件**：`packages/api/src/services/websocket.ts`

`onlineUsers` Map 只清理事件驱动的 close 事件。客户端网络异常断开（TCP RST）时可能导致内存泄漏。HEARTBEAT_INTERVAL_MS 已定义但未实现。

**修复**：添加心跳检测 + 定期清理超时连接 + Map 大小上限。

---

### 2.8 注册接口无速率限制

**现状**：登录有速率限制，但注册接口没有任何限制。攻击者可无限次尝试注册（暴力生成用户名/邮箱）。

**修复**：为注册接口添加独立速率限制（如每分钟 3 次）。

---

### 2.9 通知服务完全重复

**文件**：`packages/api/src/services/notification.ts` vs `packages/web/src/lib/notification.ts`

ServerChanNotifier、FCMNotifier、NotificationService 几乎逐行相同。

**修复**：提取为独立的 @shiguangjian/notification 包。

---

### 2.10 用户信息过度泄露

**现状**：登录接口返回 email，`/me` 接口返回 email + inviteCode。邀请码（4 位）被用于用户发现机制，但可在多个 API 中被枚举。

**修复**：登录响应移除 email；邀请码增至 8-12 位；个人资料页提供"复制"按钮而非明文展示。

---

### 2.11 用户资料更新存在 IDOR 风险

**文件**：`packages/web/src/app/api/users/me/route.ts`

```typescript
const user = await prisma.user.update({ where: { id }, data: body });
```

直接将 body 传入 Prisma，虽有限定 schema 但仍存在字段注入风险。

**修复**：显式构建更新对象，仅允许白名单字段。

---

### 2.12 Prisma 客户端实例化存在风险

**现状**：`packages/web/src/lib/prisma.ts` 用 globalThis 单例，但 Next.js Edge Runtime 不支持；两个包各自持有独立 Prisma 实例。

**修复**：API 服务中使用 Prisma 连接池配置 + SIGTERM 时 disconnect；Next.js 侧考虑移除 Prisma 改用独立 API 服务。

## 五、三方交叉讨论：共识与分歧

### 三方一致共识

1. **私钥 localStorage 存储是最危险的问题**——三位审查者均将其列为 Critical。这直接否定了产品"端到端加密"的核心卖点。
2. **双 API 架构需要统一**——架构和前端审查者一致认为 Fastify 和 Next.js Route Handlers 不应并存。
3. **JWT 密钥需要轮换**——安全审查者指出密钥弱，架构审查者补充了认证分散的问题。
4. **CORS 配置需收紧**——安全审查者和架构审查者均指出了这个问题。

### 讨论要点

**关于私钥存储的修复优先级**：
- 安全审查者主张"密码派生密钥加密存储"（PBKDF2 + AES-GCM）
- 前端审查者主张"仅内存存活，刷新后重新获取"
- 架构审查者支持前端方案，认为服务端重新生成密钥对的开销极低
- **结论**：优先采用前端方案（内存存活），中长期引入密码派生加密

**关于双 API 的去留**：
- 架构审查者倾向保留 Fastify（更适合 WebSocket 和高并发）
- 前端审查者倾向移除 Fastify（简化部署，利用 Vercel 原生能力）
- 安全审查者认为两者皆可，关键是统一认证逻辑
- **结论**：取决于部署目标——Vercel 部署选方案 B，自有服务器选方案 A

**关于邀请码长度**：
- 安全审查者指出 4 位码空间仅 167 万，易碰撞
- 架构审查者指出循环查询在用户量大时成为性能瓶颈
- **结论**：增至 8 位（36^8 约 2.8 万亿），同时改用批量预生成而非循环查询
