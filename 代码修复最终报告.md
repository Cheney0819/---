## 二、审查者之间讨论纪要

### 讨论 1：私钥存储方案
- **Alpha(Kant)**：PBKDF2 10 万次迭代在移动端可能卡顿
- **Beta(Sagan)**：建议可配置迭代次数，但当前 10 万是 OWASP 推荐值
- **Gamma(Helmholtz)**：编译验证通过，确认 `crypto.randomBytes` 在 API Routes（Node.js 环境）中可用，无类型错误
- **结论**：保持 10 万，后续可根据实际性能调整

### 讨论 2：JWT claims 对齐
- **Beta(Sagan)**：API 端和 Web 端使用不同 JWT 库（@fastify/jwt vs jose），claims 必须各自内部一致
- **Alpha(Kant)**：两边各自验证自己的 claims 不会互相干扰，因为 Fastify 和 jose 的 verify 逻辑独立
- **Gamma(Helmholtz)**：tsc 编译验证通过，无类型错误
- **结论**：API 端 `issuer='shiguangjian', audience='shiguangjian-api'`，Web 端 `issuer='shiguangjian', audience='shiguangjian-web'`，各自验证自己的 claims，互不干扰

### 讨论 3：注册限流阈值
- **Alpha(Kant)**：每分钟 3 次太严格，用户可能输错密码重试
- **Beta(Sagan)**：建议放宽到 5 次
- **Gamma(Helmholtz)**：5 次/分钟在 TypeScript 类型系统中无歧义，不会引发竞态
- **结论**：采纳 Sagan 建议，改为 5 次/分钟

### 讨论 4：CORS 本地地址
- **Beta(Sagan)**：只配了 localhost 没配 127.0.0.1，Vite/HMR 可能用 127.0.0.1
- **Alpha(Kant)**：补充后确认 localhost 和 127.0.0.1 都覆盖，开发环境无问题
- **结论**：补充 127.0.0.1:3000 和 127.0.0.1:3001

### 讨论 5：旧私钥兼容性
- **Beta(Sagan)**：新用户加密存储后，老用户的旧私钥（未加密）无法读取
- **Alpha(Kant)**：`getPrivateKey` 的回退逻辑是合理的——先解密，失败后直接解析
- **Gamma(Helmholtz)**：回退逻辑在类型系统中安全，旧 JWK 格式和新格式兼容
- **结论**：`getPrivateKey` 添加回退逻辑——先尝试解密，失败后直接解析 JWK

### 讨论 6：配对排序
- **Alpha(Kant)**：`userId < partnerId` 字符串比较在 cuid 场景下排序不可预测
- **Beta(Sagan)**：`Buffer.compare` 是二进制安全的，能正确处理任意 UUID/cuid
- **Gamma(Helmholtz)**：`Buffer` 类型在 Node.js API Routes 中编译通过
- **结论**：改用 `Buffer.compare(Buffer.from(userId), Buffer.from(partnerId))`

### 讨论 7：登出顺序（Kant 发现）
- **Alpha(Kant)**：先 `set()` 再 `removeItem()`，Zustand persist 插件可能将已写入的值重新同步回 localStorage
- **Beta(Sagan)**：同意，这是 Zustand persist 的行为特性
- **Gamma(Helmholtz)**：修正顺序后 tsc 编译无变化
- **结论**：调整为先 `removeItem` 再 `set`

### 讨论 8：z.any() 类型兼容（Helmholtz 发现）
- **Gamma(Helmholtz)**：`z.record(z.unknown())` 返回 `Record<string, unknown>` 无法直接赋给 Prisma JSON 字段
- **Alpha(Kant)**：需要类型断言
- **Beta(Sagan)**：安全，`record(z.unknown())` 仍保留了输入校验，`as any` 仅用于 Prisma 类型兼容性
- **结论**：替换为 `z.record(z.unknown())` + `(body.triggerCondition ?? null) as any`

### 讨论 9：Poll 端点缺少 POST handler（Kant 发现）
- **Alpha(Kant)**：`useRealtimeChat.ts` 的 typing 指示器发 POST 请求但端点只支持 GET，导致 typing 功能完全失效
- **Beta(Sagan)**：这是功能性 bug，不是安全问题，但影响用户体验
- **Gamma(Helmholtz)**：添加 POST handler 后 tsc 编译通过
- **结论**：已添加 POST handler

### 讨论 10：CORS 环境变量化（上线适配）
- **调度决策**：硬编码 localhost 无法用于生产部署
- **方案**：通过 `ALLOWED_ORIGINS` 环境变量驱动，逗号分隔域名列表
- **开发**：不配环境变量 → 默认 localhost 4 个端口
- **生产**：`.env` 中配 `ALLOWED_ORIGINS=https://shiguangjian.app,https://www.shiguangjian.app`
- **结论**：已实现，无需改代码，部署时只改环境变量
